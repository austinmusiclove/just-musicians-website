Questions
    Should creating application allow you to create events in the process of selecting dates? If so how do we avoid duplicates?
        Not on free version.
        Create them as recurring events with a name, location, description and dates. You can edit them on the my events page in recurring events tab
        Editing a recurring event will update all existing at once
        You can also see individual events for the recurring event in my events and can edit them as normal and those edits will only apply to one date.
        A recurring event cannot have two instances on the same day
        You can add dates to recurring event and it will automatically create individual events for each date
    Should you be able to have an application for individual events?
        No, that would be Proposal
    Should musicians be able to withdraw application?
        Yes, you should be able to see My Applications and withdraw or update
    Should you set show dates in application or allow musicians to just provide availability info for all dates?
        Free version: neither; you can invite an applicant to give you availability info on one event at a time (request for proposal)
        Paid version: application has existing events and applicants get notified to fill out availability when new events are added
    Can inquiry post type be eliminated?
        Yes
    How should availability data be stored? Per application or with event or neither?
        Per event/listing
        Proposal post type that contains event, listing, availability, draw, quote; proposal can be created when requested by venue or when application is filled out
    Does application need to be connected to a specific location?
        No but it needs a description of what the application is for
    What should we do with exisitng send inquiry?
        Send inquiry to existing event results in creating Proposal with inquiry status
        Change drop down to a modal that asks if you want to invite them to an existing event or create a new event
        Make sure to send people to event page to manage their event after they make a new one
    Should Proposal request go to messages?
        No
    Should Offer be in messages?
        No
    Should Proposal response be in messages?
        No
    Should availability be an advanced feature or just collecting quotes and draw or all free?
        Getting availability/proposals for many people at once should be advanced but requesting one at a time is possible
    When should messages be initiated?
        Only when buyer initiates; proposal and offer can contain messages and should be enough until buyer wants to open conversation
    How should Proposals and Offers appear in messages
        Leave them out of messages for now but it could be nice in the future to add them
    Can the flow work with Proposal but not Offer?
        Yes that should be enough if buyers can see all messages for an event just like current inquiry convo groups; from there they can see event details and send offers
    What happens when Offer or Proposal is accepted?
        Show event as booked in My Gigs page for musicians
        Need to ask buyer if event should change to booked
        Need email confirmations on both sides
        Send out calendar invites for event (somehow allow the buyer to set the description details with info about parking and everything else)
        Allows musician to send message to buyer
        After event has passed, both parties get notified to review
    Should conversations be per event?
        No; no offer and proposal special message types
        Group conversations by event using query var, but there is only one unique convo for a unique group of convo participants
    Should there be conversations grouped by event?
        Yes and there should be a way to filter conversations by event on main messages page
    Can buyer send message to any listing?
        Only applicants, or those with Proposals that have been replied to
        Maybe as paid feature
    Should you lose ability to access or send messages if you don't meet conditions?
        Buyers can always message an applicant listing or proposal listing
        Listings can always message a buyer who booked them for an upcoming event or responded to their proposal
        Listings can only message if they have an open conversation started by the buyer
        Once conversation is open between a buyer and listing, it is open
        Buyers can silence conversation to avoid getting too many unwanted messages from listing
    What should happen to inquiry messages?
        Leave them and never use them again; then consider removing all trace of them later
    When should users be prompted to review?
        A day after an event ends it should send an email to invite to review and in event list/calendar it should have a button to review past events if they had an accepted offer
